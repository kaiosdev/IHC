## TP3-IHC - Parte 1: Imersão e Definição

### 1. Briefing com o Cliente

O primeiro passo do trabalho foi a realização de um *briefing* com o cliente para identificar e definir o escopo da demanda.

**Cliente (Referência Técnica):** Professor Vitor Celso, Doutor em Farmacologia e docente da UFAM.
* **Contexto:** O professor atua como orientador em diversos programas acadêmicos (PIBIC, PIBITI, Pibex, Passe, Pós-Graduação) e identifica uma lacuna nos sistemas atuais da universidade, que não oferecem ferramentas para o acompanhamento e gestão da evolução desses projetos.

---

### Alinhamento de Briefing

Abaixo estão as informações coletadas durante a conversa com o cliente:

* **Funcionalidade esperada**
    * O cliente espera uma solução "prática, dinâmica e simples".
    * A funcionalidade principal é o **acompanhamento da evolução dos projetos** de pesquisa e extensão (PIBIC, Passe, Pibex, etc.).
    * Deve permitir a organização prática e concisa dos projetos, seguindo o princípio de "menos é mais".
    * Deseja um sistema que envie "lembretes" e ajude no gerenciamento de tarefas e progresso, algo que o Ecampus e o Sigele não fazem.

* **Para quem**
    * **Primariamente:** Para o professor/orientador. O professor Vítor Celso enfatizou que, "no primeiro momento", a ferramenta é para ele organizar seus projetos e orientandos.
    * **Secundariamente:** Poderia ter uma interface para o *orientando* (aluno), onde ele receberia demandas, orientações e atualizações do professor.

* **Cenário**
    * O aplicativo deve ter portabilidade, permitindo o uso "de qualquer lugar", seja no computador de casa ou no da UFAM.
    * O cenário de uso é a gestão de múltiplos projetos acadêmicos, que atualmente é feita de forma manual e adaptada (ex: "planilha toda formatada no Excel").

* **Problemas**
    * Os sistemas atuais (Ecampus/Sigele) "deixam muito a desejar", servindo apenas para submissão e aprovação, mas não para o "desenvolvimento" ou acompanhamento do projeto.
    * **Falta de especificidade:** O professor usa ferramentas genéricas (Trello, Notion), mas sente falta de "especificidade para o assunto UFAM". Ele precisa adaptar tudo à "linguagem que a UFAM tem".
    * **Perda de tempo:** Ele precisa criar manualmente "cabeçalhos" e estruturas para cada tipo de projeto (PIBIC, Pibex) e para cada aluno.

* **Prioridades**
    * A maior prioridade é ter algo "nativo" da UFAM, que já venha "pré-setado" com as características e linguagens dos editais (PIBIC, PIBITI, etc.).
    * A simplicidade e a capacidade de acompanhar a evolução real do projeto são mais importantes que um excesso de funções.

* **Fontes de informação**
    * Os **editais** (publicados pelas pró-reitorias) são as fontes primárias para as regras de submissão.
    * O aplicativo deve focar no "manejo depois que é aprovado".
    * Uma funcionalidade sugerida foi um "compêndio com a normativa legal" (leis, portarias) relevante para os projetos.

* **Integrações**
    * **Ecampus:** Uma integração direta para atualizar o Ecampus e o aplicativo simultaneamente seria "muito melhor".
    * **Pró-Reitorias (Agregador de Editais):** O cliente sugeriu uma integração que buscasse e centralizasse todos os editais abertos (Proex, Propesp, etc.) em um único menu, o que seria "maravilhoso".
    * **Pós-Graduação:** O sistema também deve considerar a gestão de orientandos de mestrado.

* **Padrão prévio (Interface)**
    * O cliente imaginou uma interface com:
        1.  Uma "barra de atualizações" ou menu listando os editais mais novos e abertos.
        2.  "Botões" para selecionar o tipo de projeto (PIBIC, Passe, Pibex, etc.).
        3.  Ao clicar, o professor veria seus projetos, dados e a "organização de forma integrada".

* **Contato técnico**
    * Atualmente, para resolver dúvidas técnicas (ex: sobre um edital), é preciso contactar as Pró-Reitorias "uma a uma", o que é um processo difícil.
